Design Spec
RUSH Icon & Logo
v1.0 2026-07-08 Concept A · Rushing Caret
01Context

The app's favicon was still the unmodified Vite starter icon — a purple abstract mark with no connection to RUSH's actual identity (near-black ink background, amber #F5A623 signal colour, JetBrains Mono, and the glowing blinking caret that is already the app's signature visual). The header also had no icon at all, just the plain-text "RUSH" wordmark.

Three concepts were sketched and compared at real favicon sizes (32px/16px) and mocked into the actual header pill during a brainstorming session. Concept A — "Rushing Caret" — was chosen.

02Decision

The app's own blinking caret, mid-sprint, trailing three shrinking, fading marks behind it. Reuses the one visual RUSH already made distinctive instead of inventing an unrelated symbol.

Rejected alternatives

  • Forward chevron (a bold >) — reads too close to a generic play/next-track glyph; not specific enough to RUSH.
  • Ascending bars (WPM meter) — a fine "speed/stats" mark, but has less to do with the actual act of typing than the caret does.
03Mark spec

Two renderings of the same geometry, scaled to context:

1. Favicon / app-icon tile — public/favicon.svg, 40×40 viewBox

#0B0F14

caret:  x=22 y=8  w=4 h=24 rx=2       fill=#F5A623
trail1: x=14 y=18.2 w=5   h=3.6 rx=1.8  opacity=.90
trail2: x=8  y=18.2 w=4   h=3.6 rx=1.8  opacity=.55
trail3: x=4  y=18.2 w=2.4 h=3.6 rx=1.8  opacity=.30
glow:   drop-shadow(0 0 5px rgba(245,166,35,.55))

2. Header mark — bare, 24×24 viewBox (matches sibling icons)

No background tile — just the caret + trail, re-expressed on the same 24×24 viewBox used by every other icon in src/components/icons/. Rendered with fill="currentColor" so it inherits whatever colour class wraps it (here, the header's existing text-signal) — no hardcoded hex in the component.

05Out of scope
  • No README/social-preview logo — scope was explicitly favicon + header only.
  • No PNG/ICO/apple-touch-icon raster fallback. No SVG rasterizer is available on this machine, and evergreen browsers already support SVG favicons directly (the project ships one this way today). A small follow-up once a rasterizer is available if iOS home-screen icons matter later.
  • public/icons.svg (the unrelated leftover social-icon sprite sheet) is untouched — unused, out of scope for this change.
06Verification
Visual check (browser tab + header lockup) plus the existing validation suite — no new tooling needed for this change.
  • Dev server: browser tab shows the new favicon; header shows mark + "RUSH" as one lockup, colour matching.
  • pnpm validate (lint / type-check / prettier) and pnpm build both pass.
  • No new hardcoded hex in Header/index.tsx — mark colour comes from currentColor + the existing text-signal utility.
